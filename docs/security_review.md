## 3. Аутентификация и управление доступом

### 3.1 Механизм входа
- Вход осуществляется по **номеру телефона + пароль**
- Доступ разрешён только номерам из **белого списка (whitelist)**, который ведётся администратором системы вручную
- Номера телефонов, не внесённые в whitelist, получают отказ на этапе проверки телефона — **до ввода пароля**
- Пароли хэшируются алгоритмом **bcrypt** (соль генерируется автоматически)

### 3.2 Управление сессиями
- Сессии хранятся в **зашифрованном cookie** (алгоритм: AES через itsdangerous/HMAC-SHA256)
- Ключ шифрования (`SECRET_KEY`) — 64-символьная случайная строка, хранится в переменных окружения, **не в коде**
- Срок жизни сессии: **7 дней**
- Инвалидация сессии: при сбросе пароля администратором версия сессии (`session_version`) увеличивается — все активные сессии пользователя немедленно аннулируются
- Cookie-параметры: `SameSite=Lax`, `HttpOnly=True`, `Secure=True` (в production)

### 3.3 Биометрическая аутентификация (опционально)
- Поддерживается **WebAuthn** (Face ID / Touch ID) как второй фактор
- Биометрические данные **не передаются и не хранятся** на сервере — хранится только публичный ключ устройства
- Стандарт: FIDO2 / WebAuthn Level 2

### 3.4 Уровни доступа
| Роль | Возможности |
|---|---|
| Пользователь | Просмотр и управление своими задачами и проектами |
| Администратор | Управление доступом (whitelist), сброс паролей |
| Системный (Месмер Д.) | Аудит-лог активности пользователей |

- Excel-файлы для импорта проектов (макс. 10 МБ)
- Допустимые форматы файлов проверяются на стороне сервера по расширению

## 5. Внешние сервисы и передача данных

### 5.1 Перечень внешних сервисов

| Сервис | Назначение | Передаваемые данные | Обязательность |
|---|---|---|---|
| **Brevo** (Sendinblue) | Email-уведомления | Имя получателя, тема, текст письма | Опционально |
| **Groq API** | ИИ-ассистент (Llama 3.3) | Обезличенные рабочие данные о проектах | Опционально |
| **Anthropic API** | ИИ-ассистент (Claude) | Обезличенные рабочие данные о проектах | Опционально |
| **DeepSeek API** | ИИ-ассистент | Обезличенные рабочие данные о проектах | Опционально |
| **Cloudinary** | Хранение фотографий | Фото объектов, ВПК, задач | Опционально |
| **Sentry** | Логирование ошибок | Технические стектрейсы, без данных пользователей | Опционально |

**Важно:** Все внешние сервисы **опциональны**. При размещении в корпоративной инфраструктуре:
- ИИ-ассистент можно отключить (убрать API-ключи)
- Email заменить на корпоративный SMTP-сервер
- Cloudinary заменить на корпоративное хранилище (S3-совместимое)
- Sentry заменить на self-hosted Glitchtip или отключить

### 5.2 Передача данных наружу
- Все внешние запросы выполняются **только с серверной стороны** (не из браузера пользователя)
- Используется протокол **HTTPS/TLS** для всех внешних соединений
- Персональные данные сотрудников в ИИ-сервисы **не передаются** — только агрегированная рабочая информация (количество проектов, даты, статусы)

## 6. Меры защиты от атак

### 6.1 CSRF (межсайтовая подделка запросов)
- Все POST-формы защищены **CSRF-токенами** (256-битный случайный токен)
- Токен генерируется на сервере, хранится в зашифрованной сессии
- Проверяется при каждом POST-запросе через `CSRFMiddleware`
- Исключения: только `/api/` и `/login/` (публичные endpoints)

### 6.2 Rate Limiting (ограничение частоты запросов)
- Вход по телефону: **20 запросов/минуту**
- Ввод пароля: **5 запросов/минуту** (защита от брутфорса)
- Создание пароля: **10 запросов/минуту**
- Реализовано через библиотеку slowapi (IP-based)

### 6.3 XSS (межсайтовый скриптинг)
- Все данные экранируются шаблонизатором Jinja2 по умолчанию
- Явное использование `| safe` отсутствует в пользовательских данных
- В JavaScript используется функция `escHtml()` для данных из API

### 6.4 SQL-инъекции
- Все запросы к БД выполняются через **SQLAlchemy ORM** с параметризацией
- Прямые SQL-запросы в коде отсутствуют

### 6.5 Загрузка файлов
- Проверка расширения файла на сервере
- Ограничение размера: Excel 10 МБ, фото 20 МБ, документы 50 МБ
- Имена файлов санируются (удаляются спецсимволы)
- Файлы сохраняются вне корня веб-сервера (`/static/uploads/`)

## 7. Журналирование и аудит

### 7.1 Аудит-лог активности
- Фиксируется каждое посещение страницы авторизованным пользователем
- Хранится: имя пользователя, номер телефона, URL страницы, IP-адрес, время
- Доступ к аудит-логу: **только системный администратор** (проверка по номеру телефона)
- Реализовано через `AuditMiddleware` на уровне ASGI

### 7.2 Логирование ошибок
- Технические ошибки: логируются через стандартный Python `logging`
- Критические ошибки на production: отправляются в Sentry (при наличии DSN)
- Sentry настроен с `send_default_pii=False` — персональные данные не передаются

---

## 8. Сетевая безопасность

- Приложение работает за **nginx** (reverse proxy)
- Внешний доступ: только порты **80/443**
- Все запросы перенаправляются на **HTTPS**
- Cookie-флаг `Secure=True` — cookie не передаётся по HTTP
- Cookie-флаг `SameSite=Lax` — защита от CSRF через сторонние сайты
- Статические файлы раздаёт nginx (не Python-приложение)

## 10. Рекомендации при размещении в корпоративной инфраструктуре

1. Разместить приложение во **внутренней сети** (интранет), без доступа из интернета
2. Использовать **корпоративный SMTP** вместо Brevo
3. Хранить все `SECRET_KEY` и ключи API в **корпоративном хранилище секретов** (Vault или аналог)
4. Настроить **регулярный backup** PostgreSQL (ежедневно, хранение 30 дней)
5. Настроить мониторинг доступности (uptime)
6. При необходимости — отключить ИИ-функции (удалить GROQ_API_KEY, ANTHROPIC_API_KEY)
